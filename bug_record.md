# bug_record

## 短答案任务回答后继续生成多余 token

### 现象

在 stage1 eval / 服务测试中，模型经常会先输出正确短答案，例如：

- `苏州`
- `北京`
- `临湖宾馆`

但随后不会立即停止，而是继续生成额外文本，例如下一段问答、解释性内容或其它无关 token，直到遇到新的停止条件或 `max_new_tokens`。

### 当前判断

1. 这不代表答案本身错误，很多情况下首个短答案是正确的。
2. 问题在于回答后的停止边界不稳定，模型没有及时生成 EOS。
3. 当前输出里还可能混入后续 continuation，导致最终展示结果看起来很乱。

### 根因分析方向

1. 早期版本训练数据中 answer 末尾缺少 EOS，导致模型没有充分学会“短答案后立即结束”。
2. `问题：...\n答案：` 这种模板可能触发模型进入连续 QA 续写模式，答完一题后继续生成下一题。
3. frozen backbone 下，原模型的续写习惯保留较多，停止边界更依赖 prompt 和监督格式。

### 当前临时缓解方式

由于当前任务的答案通常只有一句话/一个短实体，展示层可以先做临时截断：

- 按 `\n` 取第一行
- 仅保留第一句/首段短答案

这属于临时输出清洗，不是根本修复。

### 根本修复方向

1. 训练数据 answer 末尾加入 EOS。
2. 保持训练 / 推理 prompt 一致。
3. 后续尝试更强约束模板，例如：

```text
根据当前历史信息，回答下列问题。
请只输出最终答案，不要解释。
问题：{question}
回答：
```

4. 继续观察新 checkpoint 是否能显著提高短答案后 EOS 的触发概率。
