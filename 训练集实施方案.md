# 训练集实施方案

本文档用于单独讨论：

- 第一阶段训练集怎么生成
- 第一阶段测试集怎么设计
- 第二阶段目前存在哪些设计问题
- 生成后的数据集如何存储、查看、抽检

目标不是立即把所有阶段一次性定死，而是先把**第一阶段可落地的数据集方案**做清楚，并且保证生成后用户能直接看到样本长什么样。

---

## 1. 设计目标

本项目当前训练路线是：

- 冻结 backbone（当前主用 `Qwen3.5-7B`）
- 仅训练 Titans memory
- 采用 MAC（Memory as Context）思路

因此数据集的核心目标不是一般 SFT，而是让模型学会：

1. 在 history 中识别并写入重要信息
2. 在 question 阶段检索出相关记忆
3. 让冻结 backbone 利用 memory context 回答问题

---

## 2. 第一阶段总体原则

第一阶段不追求复杂真实推理，重点是打通“写入-检索-回答”闭环。

因此第一阶段采用：

- **程序批量生成**
- **结构化可控任务**
- **答案唯一且易校验**
- **单独建设测试集**

第一阶段建议拆成两个集合：

1. **训练集（train）**
2. **测试集（eval）**

其中测试集又分为：

- `eval_structured`：结构化记忆题
- `eval_itinerary`：行程对话记忆题

---

## 3. 第一阶段训练集：生成思路

### 3.1 不是直接改现成公开训练集

第一阶段不建议从通用 SFT 数据集（如 Alpaca / ShareGPT）修改得到。

原因：

- 长程记忆监督不够硬
- 很难稳定控制干扰项
- 很难构造“更新覆盖”任务
- 不利于做阶段性 ablation

因此，第一阶段更适合：

- 先构建**基础素材池**
- 再程序化生成训练样本

---

## 4. 第一阶段训练集：基础素材池

第一阶段建议先有一个“素材池”层，而不是直接拼训练样本。

### 4.1 必备素材池

#### A. 人名池

用于生成：

- 用户名称
- 客户名称
- 联系人名称

建议第一版采用两种方式兼容：

1. **程序生成中文姓名**（默认）
2. **可选外部名字文件**（增强）

默认可采用：

- 姓氏池：常见中文姓氏
- 名字字池：常见中文名字字

组合生成几百到几千个名字。

#### B. 城市池

例如：

- 北京、上海、苏州、杭州、深圳、广州、南京、成都、武汉

#### C. 公司 / 客户池

例如：

- 华辰科技
- 云岚数据
- 安拓物流
- 恒川制造

#### D. 项目池

例如：

- 凤凰项目
- 海潮计划
- 北极星升级

#### E. 值生成器

用于自动生成：

- 工号
- 电话尾号
- 日期
- 时间
- 航班号
- 酒店名

#### F. 噪音模板池

例如：

- `{name} 今天参加了周会。`
- `团队计划下周继续推进联调。`
- `项目讨论主要围绕预算和进度。`

---

## 5. 第一阶段训练集：任务类型

建议第一阶段训练集只做三类任务。

### 5.1 单事实召回 `single_kv`

示例：

- history 中：`陈默的工号是 A17-9421。`
- question：`陈默的工号是什么？`
- answer：`A17-9421`

训练目标：

- 基础写入与精确读取

### 5.2 多实体同属性干扰 `multi_kv`

示例：

- 多个人都有工号 / 城市 / 联系方式
- 只问其中一个人的指定属性

训练目标：

- 学会实体-属性绑定
- 防止只记住“最近一个值”

### 5.3 更新覆盖 `update_kv`

示例：

- `李想原定周二去上海。`
- `后来改成周三去苏州。`
- question：`李想现在要去哪里？`
- answer：`苏州`

训练目标：

- 学会保留最终有效事实
- 不被旧信息误导

---

## 6. 第一阶段训练集：推荐样本结构

建议每条训练样本按 chunk 组织：

```json
{
  "history_chunks": [
    "...",
    "..."
  ],
  "question_chunk": "...",
  "answer": "...",
  "meta": {
    "task_type": "single_kv",
    "num_history_chunks": 8,
    "target_chunk_index": 5,
    "num_distractors": 6,
    "has_update": false
  }
}
```

说明：

- `history_chunks`：模型写 memory 的输入
- `question_chunk`：触发 retrieval 的问题
- `answer`：监督目标
- `meta`：用于分析和抽检

---

## 7. 第一阶段测试集：必须单独建设

第一阶段除了训练集，还必须单独做测试集。

原因：

- 训练集只验证能否学习
- 测试集要验证是否真的形成稳定记忆能力

第一阶段建议做两类测试集。

### 7.1 结构化测试集 `eval_structured`

与训练集同任务类型，但样本不重合。

作用：

- 测试单事实召回能力
- 测试多实体干扰下的检索能力
- 测试更新覆盖能力

### 7.2 行程对话测试集 `eval_itinerary`

这是第一阶段非常重要的一部分。

目标：

- 用户聊了一段行程后
- 模型能够记住：
  - 用户名称
  - 客户名称
  - 行程安排
  - 更新后的最终安排

这部分更接近真实使用场景。

---

## 8. 行程对话测试集：建议设计

### 8.1 目标字段

建议每条样本覆盖至少这些信息：

- 用户名称
- 客户名称
- 日期 / 星期
- 城市 / 地点
- 酒店 / 航班 / 高铁
- 会议安排
- 更新后的最终安排

### 8.2 问题类型

建议覆盖以下 5 类：

1. 用户身份记忆
2. 客户记忆
3. 时间记忆
4. 地点记忆
5. 更新记忆

### 8.3 推荐样本结构

建议一条对话配多个问题：

```json
{
  "dialogue": [
    {"role": "user", "text": "我叫陈默，下周要去上海见客户王总。"},
    {"role": "user", "text": "周二上午先到浦东，下午两点和王总开会。"},
    {"role": "user", "text": "周三改去苏州工厂，酒店订在金鸡湖附近。"}
  ],
  "qa_pairs": [
    {"question": "用户叫什么名字？", "answer": "陈默"},
    {"question": "客户叫什么？", "answer": "王总"},
    {"question": "周三去哪里？", "answer": "苏州工厂"},
    {"question": "酒店订在哪里？", "answer": "金鸡湖附近"}
  ],
  "meta": {
    "sample_type": "itinerary_dialogue",
    "has_update": true
  }
}
```

## 9. 第二阶段目前的主要问题

第二阶段大概率不再只是结构化 KV 召回，而会进入更接近真实长对话的阶段。

当前主要问题有：

### 9.1 分布跳跃风险

如果第一阶段全是模板化样本，第二阶段直接切到高度自然化对话，分布跳跃会过大。

### 9.2 query 表达复杂化

第二阶段问题会出现：

- 指代
- 省略
- 改写
- 多条件过滤

### 9.3 更新逻辑更复杂

真实场景里可能多次更新：

- 改时间
- 改地点
- 改客户
- 改酒店

### 9.4 噪音控制问题

第二阶段噪音如果过强，可能使 memory 学习到不稳定行为。

### 9.5 评测维度不足

第二阶段不能只看总体准确率，还要分字段看：

- 用户名
- 客户名
- 时间
- 地点
- 更新题

## 10. 存储格式：建议怎么保存

你特别关心“先生成，让你知道数据集长什么样”，所以存储方式必须便于：

- 批量生成
- 人眼查看
- 抽样检查
- 后续训练直接读取

并且需要满足一个额外约束：

- **生成的数据集统一存放在专门目录下**
- **训练脚本默认统一从该目录加载**

因此不建议把生成数据散落到多个目录，也不建议训练脚本把路径写死在多个位置。

### 10.0 统一数据目录约束

建议统一使用：

- `data/generated/stage1/`

目录示例：

```text
data/
└── generated/
    └── stage1/
        ├── train.jsonl
        ├── eval_structured.jsonl
        ├── eval_itinerary.jsonl
        ├── preview.md
        └── stats.json
```

建议约定：

- 生成脚本默认输出到 `data/generated/stage1/`
- 训练脚本默认从 `data/generated/stage1/` 读取
- 若用户未显式指定路径，则都使用该默认目录

### 10.1 推荐主格式：JSONL

建议训练集和结构化测试集主格式统一用：

- `.jsonl`

原因：

- 一行一条样本
- 便于流式处理
- 便于后续直接训练读取
- 便于命令行抽样查看

建议目录：

```text
data/
└── generated/
    └── stage1/
        ├── train.jsonl
        ├── eval_structured.jsonl
        └── eval_itinerary.jsonl
```

### 10.2 推荐辅助查看格式：Markdown 预览文件

为了让你先看“数据集长什么样”，我建议每次生成时，**额外输出一个 sample preview 文件**：

- `data/generated/stage1_preview.md`

里面放：

- 训练集随机抽样若干条
- 结构化测试集抽样若干条
- 行程测试集抽样若干条

这样你不用直接读 jsonl，也能很快看出样本质量。

### 10.3 可选辅助格式：摘要统计 JSON

建议额外生成：

- `data/generated/stage1_stats.json`

更准确地建议放在统一目录下：

- `data/generated/stage1/preview.md`
- `data/generated/stage1/stats.json`

记录：

- 各任务类型数量
- chunk 数分布
- 平均 question 长度
- 更新样本比例
- retrieval 目标字段分布

## 11. 如何查看生成后的数据集

我建议第一版生成脚本默认同时产出三类结果：

1. 主数据集 `.jsonl`
2. 预览文件 `.md`
3. 统计文件 `.json`

### 11.1 你最先看的文件

建议先看：

- `data/generated/stage1/preview.md`

它应该以人类友好的格式展示，例如：

```md
## Sample 1 - single_kv

### History
1. ...
2. ...

### Question
陈默的工号是什么？

### Answer
A17-9421

### Meta
- task_type: single_kv
- target_chunk_index: 3
```

### 11.2 训练时读取的文件

训练脚本直接读取：

- `data/generated/stage1/train.jsonl`

并建议在训练脚本中保留一个默认参数，例如：

- `--data-dir data/generated/stage1`

如果用户不传该参数，则默认读取这个目录。

### 11.3 抽样检查方式

生成脚本建议支持：

- 生成固定数量样本
- 固定随机种子
- 输出前 N 条 preview

这样每次改数据规则后，你都能马上看到差异。

## 12. 我建议的数据生成流程

### 第一轮

先生成一个很小的版本用于检查：

- train: 100 条
- eval_structured: 20 条
- eval_itinerary: 10 条

只用于确认：

- schema 是否合理
- 文本是否自然
- question / answer 是否清晰
- 更新逻辑是否正确

### 第二轮

确认没问题后，再生成中等规模：

- train: 5k ~ 20k
- eval_structured: 500
- eval_itinerary: 100 ~ 300

### 第三轮

训练跑通后，再继续扩规模。

## 13. 当前建议的实现方式

建议新增脚本：

- `scripts/build_stage1_memory_data.py`

建议脚本至少支持：

- 指定训练集数量
- 指定结构化测试集数量
- 指定行程测试集数量
- 指定随机种子
- 输出 jsonl
- 输出 preview.md
- 输出 stats.json

概念上类似：

```bash

  --train-size 100 \
  --eval-structured-size 20 \
  --eval-itinerary-size 10 \
  --seed 42 \
  --output-dir data/generated/
```

## 14. 第一阶段数据集规模建议

第一阶段不建议一开始就生成超大规模数据集。

更合理的方式是分三档推进：

1. **预览集**：先看样本长什么样
2. **小规模可训练集**：先跑通训练闭环
3. **正式第一阶段集**：在路线可行后扩规模

### 14.1 预览集（人工检查）

建议规模：

- `train`: 100
- `eval_structured`: 30
- `eval_itinerary`: 20

用途：

- 检查 schema 是否合理
- 检查 question / answer 是否清晰
- 检查更新样本是否符合预期
- 检查行程对话测试是否贴近真实需求

这一档不以训练效果为目标，主要用于**看样本和改规则**。

### 14.2 小规模可训练集（闭环验证）

建议规模：

- `train`: 5,000 ~ 10,000
- `eval_structured`: 300 ~ 500
- `eval_itinerary`: 100 ~ 200

推荐默认值：

- `train`: 8,000
- `eval_structured`: 400
- `eval_itinerary`: 150

用途：

- 跑通第一版训练 baseline
- 验证 memory 是否学会基础写入与检索
- 检查 itinerary 测试上的早期表现

### 14.3 正式第一阶段集（阶段训练）

建议规模：

- `train`: 30,000 ~ 80,000
- `eval_structured`: 1,000
- `eval_itinerary`: 300 ~ 500

推荐默认值：

- `train`: 50,000
- `eval_structured`: 1,000
- `eval_itinerary`: 300

用途：

- 正式进行第一阶段训练
- 进行 memory 机制验证
- 为后续第二阶段提供 baseline

### 14.4 为什么第一阶段不建议更大

因为第一阶段的目标不是靠海量数据硬灌，而是验证：

- memory 机制是否成立
- frozen backbone + memory 接口是否成立
- 干扰 / 更新 / 行程记忆能力是否有效

如果数据过大，会增加：

- 生成成本
- 抽检难度
- 错误样本污染范围
- 训练迭代成本

因此第一阶段建议先控制在合理规模内。

## 15. 第一阶段训练集任务比例建议

如果采用正式第一阶段规模，例如 `train = 50,000`，建议任务比例为：

- `single_kv`: 40%
- `multi_kv`: 35%
- `update_kv`: 25%

对应数量示例：

- `single_kv`: 20,000
- `multi_kv`: 17,500
- `update_kv`: 12,500

### 15.1 这样分配的原因

#### `single_kv` 40%

保证基础写入与读取足够稳定，是第一阶段的根基。

#### `multi_kv` 35%

保证模型不会只学到“最近值”或“关键词匹配”，而是学到实体-属性绑定。

#### `update_kv` 25%

让模型开始具备更新覆盖能力，但第一阶段不宜压得过高，否则训练初期更容易不稳定。

---

## 16. 第一阶段生成顺序建议

建议按以下顺序推进，而不是一步到位：

### Step 1：生成预览集

输出：

- 小规模 `jsonl`
- `preview.md`
- `stats.json`

先看样本内容是否满意。

### Step 2：修正规则

根据预览结果修正：

- 模板自然度
- question 表达
- update 逻辑
- itinerary 对话质量

### Step 3：生成小规模可训练集

跑 baseline，确认：

- loss 正常下降
- eval_structured 有效果
- eval_itinerary 有初步记忆能力

### Step 4：生成正式第一阶段集

在小规模闭环成立后，再扩到正式规模。

---

## 17. 当前结论

当前第一阶段的数据集方案建议是：

1. 不依赖通用公开 SFT 数据集改造
2. 先构建基础素材池
3. 用程序批量生成训练集
4. 单独构建结构化测试集
5. 单独构建行程对话测试集
6. 存储采用：
   - `jsonl` 作为主格式
   - `preview.md` 作为人工查看格式
   - `stats.json` 作为统计摘要
7. 数据集按“三档规模”推进：
   - 预览集
   - 小规模可训练集
   - 正式第一阶段集

这样你可以先看到数据长什么样，再决定是否扩大生成规模并进入训练。
